# Git / CI-CD / AI 协作治理标准

## 1. 目的

本文档用于定义 `BTP_Integration` 团队在 `Git`、`CI/CD` 与 `AI 协作使用` 方面的基础治理标准。

目标不是一次性建立复杂制度，而是先形成团队可执行的最低治理基线，逐步升级到可持续的工程化协作模式。

## 2. 适用范围

本文标准适用于：

- 本仓库及后续团队管理的相关仓库
- 所有人工成员与 AI 角色的仓库操作行为
- 文档、脚本、配置、设计资产和项目资料的版本管理
- 未来接入的自动检查、构建、发布和质量门禁流程

## 3. 治理目标

- 所有关键变更可追踪
- 仓库结构清晰、污染可控
- 分支协作可理解、可回溯
- 提交说明清楚，便于审计和复盘
- AI 的操作边界明确，不破坏仓库秩序
- `CI/CD` 从轻量检查逐步走向标准化交付

## 4. 角色职责

| 角色 | 主要责任 |
| --- | --- |
| `大黄` | 统筹治理方向、优先级与资源 |
| `Kai` | 版本治理设计、检查、规范沉淀、教育推广 |
| `wl` | 开发落地与实施可行性 |

## 5. Git 基础规则

### 5.1 仓库边界

- 一个仓库应对应一个清晰的业务或交付边界
- 临时处理目录、解压目录、日志、中间产物不得默认纳入版本管理
- 所有应忽略内容必须写入 `.gitignore`

### 5.2 默认分支

- 默认主分支统一为 `main`
- `main` 只允许合并经过审查的内容
- 非紧急情况不建议直接在 `main` 上开发

### 5.3 分支命名规则

推荐分支前缀：

- `feature/<topic>`
- `fix/<topic>`
- `doc/<topic>`
- `chore/<topic>`
- `release/<version>`

示例：

- `feature/value-mapping-governance`
- `doc/poc-case-story-update`
- `fix/gitignore-cleanup`

### 5.4 提交规范

提交信息应简短、明确、可审计，建议格式：

```text
<type>: <summary>
```

推荐 `type`：

- `feat`
- `fix`
- `docs`
- `refactor`
- `test`
- `chore`

示例：

- `docs: add value mapping technical design`
- `chore: initialize git repository`
- `fix: exclude temp ppt artifacts from git`

### 5.5 合并规则

- 合并前应至少完成一次自检
- 涉及规则、标准、架构或对外资料的改动，建议至少一轮评审
- 大范围改动应拆分提交，避免一次提交混杂多个目的
- 不允许将无关临时文件混入业务提交

### 5.6 文档变更提交规则

- 任何 `Markdown`、`CSV`、设计说明、标准文档、导引文档、对外案例文档的变更，在内容完成后都必须纳入版本管理
- 不允许长期保留“已完成但未提交”的文档改动
- 若一次任务只涉及文档，原则上应在任务完成当次形成至少一次 `docs:` 类型提交
- 若任务同时涉及规则文档与项目文档，应确保两类内容在提交说明中可被识别，必要时拆分提交
- 提交前必须执行一次 `git status` 自检，确认没有混入无关文件
- `Kai` 应对文档类任务执行提交闭环检查，确保“已修改文档 -> 已入版 -> 已提交”链路成立

## 6. 标签与发布规则

- 需要形成阶段性基线时使用 Tag
- 推荐格式：`v<major>.<minor>.<patch>`
- 若仅为文档里程碑，可加说明，如：`v0.1.0-doc-baseline`
- 发布说明至少记录：范围、主要变更、已知风险、回退点

## 7. `.gitignore` 与仓库卫生标准

以下内容原则上不应纳入版本控制：

- 解压产物
- 临时脚本输出
- 调试日志
- 中间文本
- 大体积压缩包
- 本地 IDE 缓存
- 凭据、令牌、机密文件

每次新引入工具、流程或自动化脚本后，都应检查是否需要更新 `.gitignore`。

## 8. 周期性检查机制

### 8.1 周检

每周至少检查以下内容：

- 是否存在误提交的临时文件
- 是否有长期未合并分支
- 是否有提交信息质量明显下降
- 是否存在直接在 `main` 上频繁提交的迹象
- 是否有规则文档与实际操作脱节

### 8.2 月检

每月至少检查以下内容：

- 分支策略是否仍适合团队规模
- `.gitignore` 是否需要补充
- 是否需要新增自动化检查
- 是否存在重复性人工检查，可转为 `CI`
- AI 是否重复犯同类仓库操作问题

### 8.3 检查输出

检查结果建议形成固定模板，至少包含：

- 本期检查范围
- 发现的问题
- 风险等级
- 责任人
- 修正建议
- 截止时间

## 9. AI 仓库操作治理标准

### 9.1 AI 允许的行为

- 在明确任务范围内创建、修改、整理文件
- 初始化本地版本库
- 添加合理 `.gitignore`
- 提交结构化、可解释的变更
- 输出检查报告、治理建议、规则文档

### 9.2 AI 禁止的行为

- 无说明直接大规模改动仓库结构
- 擅自删除用户已有重要文件
- 在未评估影响时强推复杂流程
- 绕过规则直接覆盖关键分支内容
- 提交与任务无关的大量噪音文件
- 将敏感信息、密钥、凭据提交入库

### 9.3 AI 操作要求

- 每次重大修改前说明目标和范围
- 每次提交前确认变更内容与目的匹配
- 发现仓库异常时优先提示，不得静默处理
- 对规则类改动需同时更新文档
- 输出结论时要明确哪些是已完成，哪些是建议

## 10. AI 规范教育要求

`Kai` 应持续对其他 AI 进行以下教育：

- 何时应该新建分支，何时可直接在当前分支继续
- 什么样的文件不应该提交
- 提交信息怎样写才便于复盘
- 规则文档、标准文档、项目文档如何区分
- 什么情况下必须升级给 `大黄`、`AK`、`Nova`

建议把高频错误沉淀为：

- 提交前自检清单
- 常见违规示例
- 角色协作边界说明
- 仓库健康检查模板

## 11. CI/CD 分阶段路线图

### 阶段 1：基础治理

- 接入 `Git`
- 建立 `.gitignore`
- 明确分支与提交规范
- 形成周检机制

### 阶段 2：轻量自动检查

- 自动检查文件命名与目录结构
- 自动识别临时文件、日志、压缩包误提交
- 自动校验 Markdown / CSV 基础规范
- 自动输出变更摘要

### 阶段 3：质量门禁

- 对关键文档目录启用合并前检查
- 检查标准文档是否同步更新
- 对关键资产启用 reviewer 机制
- 将高风险目录纳入更严格审批

### 阶段 4：发布治理

- 建立 Tag 与里程碑机制
- 输出发布说明模板
- 明确回滚基线
- 形成项目阶段性版本快照

## 12. 最小可执行检查清单

每次准备提交前，至少确认：

- 本次变更目标是否清楚
- 是否混入无关文件
- 是否应更新 `.gitignore`
- 是否需要同步更新标准或索引文档
- 提交信息是否足够让别人看懂
- 是否需要走评审或请 `大黄` 统筹
- 若本次为文档任务，是否已经完成当次提交闭环

## 13. 当前团队建议

结合当前仓库现状，建议优先推进以下事项：

1. 保持 `main` 作为主分支
2. 将文档类更新优先规范为 `docs:` 提交
3. 逐步建立周检表，而不是一开始强上复杂流水线
4. 优先治理临时文件、压缩包、解压目录和规则文档一致性
5. 由 `Kai + Nova + 大黄` 联合推进 AI 仓库操作规范

## 14. 一句话结论

团队的 `Git / CI-CD / AI` 治理，不应只是“工具接上了”，而应让每一次变更都更可控、更清楚、更便于协作和复盘。
