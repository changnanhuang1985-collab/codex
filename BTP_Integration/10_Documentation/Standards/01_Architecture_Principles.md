# アーキテクチャ原則

## 1. 目的

本書は、`SAP BTP Integration` を利用したシステム連携において、一貫性・拡張性・保守性・セキュリティ・運用性を確保するためのアーキテクチャ原則を定義する。

## 2. 適用範囲

本標準は以下の領域に適用する。

- `Cloud Integration`
- `API Management`
- `Event Mesh`
- `Integration Advisor`
- `Open Connectors`
- 周辺システムとの連携設計

## 3. 基本原則

### 3.1 業務要件起点

- 技術選定は必ず業務要件、非機能要件、運用要件を起点とする
- 「使えるから使う」ではなく「要件に適合するから採用する」を原則とする

### 3.2 責務分離

連携基盤内の責務は明確に分離する。

- `API Layer`
  - 外部公開、利用制御、セキュリティ、レート制御、契約管理を担う
- `Integration Layer`
  - 変換、ルーティング、オーケストレーション、エラー制御を担う
- `Event Layer`
  - 非同期通知、疎結合連携、イベント配送を担う

### 3.3 疎結合

- 連携先の内部仕様に過度に依存しない
- フロントシステムとバックエンドシステムの直接結合を避ける
- インタフェース契約は明文化し、内部実装とは分離する

### 3.4 再利用性

- 共通ロジック、共通変換、共通エラーハンドリングは再利用を前提とする
- 同一要件を複数 `iFlow` で重複実装しない

### 3.5 可観測性

- すべての本番連携は監視可能でなければならない
- 障害時に追跡可能な識別子、ログ、メッセージ履歴を保持する

### 3.6 セキュリティ・バイ・デザイン

- 認証・認可・通信保護・秘密情報管理を設計初期から考慮する
- 開発後の後付け対応を避ける

### 3.7 運用可能性

- 本番稼働後の監視、再処理、調査、変更管理までを含めて設計する
- 開発容易性のみを優先しない

## 4. 統合スタイルの適用原則

### 4.1 Process Integration

適用条件：
- 複数システムにまたがる業務プロセス連携が必要
- 順序制御、分岐、集約、状態遷移が必要

不適用条件：
- 単純データ複製のみ
- 単一 API 呼び出しのみ

### 4.2 Data Integration

適用条件：
- マスタ・トランザクションデータの複製・同期が主目的
- 変換はあるが複雑なプロセス制御は不要

### 4.3 API-based Integration

適用条件：
- システム能力を標準化された契約で公開したい
- 利用者管理、認証、流量制御が必要

### 4.4 Event-driven Integration

適用条件：
- リアルタイム性が必要
- 疎結合な通知連携が必要
- 将来的なサブスクライバ追加が想定される

### 4.5 User Integration

- 原則として `Integration Suite` の主担当領域ではない
- ユーザポータル・ワークスペース統合は `SAP Build Work Zone` 等の適切なサービスを優先検討する

### 4.6 B2B Integration

適用条件：
- 企業間メッセージ交換
- EDI/業界標準との対応
- パートナ別マッピング管理が必要

## 5. 推奨レイヤ構成

```text
Consumer / External Apps
        |
   API Management
        |
 Cloud Integration
        |
Backend / SaaS / Partner Systems
        |
   Event Mesh (when needed)
```

## 6. アーキテクチャ設計時の必須確認項目

- 連携目的は何か
- 同期 / 非同期どちらか
- データのシステムオブレコードはどこか
- 再送は必要か
- API として公開すべきか
- イベント連携の方が適切ではないか
- 監視対象と運用責任者は誰か
- セキュリティ方式は何か
- 将来の利用者追加に耐えられるか

## 7. アンチパターン

- `API Management` を使わずに外部公開を乱立させる
- 1 本の `iFlow` に複数業務責務を詰め込む
- スクリプトで何でも実装する
- ログや監視要件が未定義のまま本番化する
- イベント向き要件を同期連携で無理に実装する
- 設計文書なしで開発を先行する

## 8. レビュー観点

- レイヤ責務は明確か
- 選択した統合スタイルに妥当性があるか
- 将来変更時の影響範囲は限定されているか
- 非機能要件が具体化されているか
- 運用設計まで含めて整合しているか
