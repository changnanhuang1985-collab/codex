# CPI 品質チェックリスト

## 1. 目的

本チェックリストは、`SAP Integration Suite / Cloud Integration (CPI)` を中心とした連携案件において、各フェーズで品質を守るための確認観点を標準化することを目的とする。

本資料は `AK` が技術観点を整理し、`田内` が実行可能なチェック項目として構造化したものである。

想定用途：

- 要件整理時の抜け漏れ防止
- 設計レビュー
- 開発レビュー
- テスト前確認
- 本番移行前確認
- 初期運用時の品質確認

## 2. 基本方針

- 品質はテスト工程だけで作るものではなく、各工程で作り込む
- `CPI` の品質は、設計、実装、試験、監視、運用準備まで一体で担保する
- 各フェーズで「次工程に渡してよい状態か」を判断する

## 3. フェーズ別チェックリスト

## 3.1 要件整理フェーズ

### 業務要件

- 連携目的が明確になっている
- 業務シナリオが説明できる
- 送信元 / 送信先システムが明確
- 同期 / 非同期が定義されている
- トリガ方式が定義されている
- データ鮮度要件が明確
- 障害時の業務影響が整理されている

### 連携方式

- `CPI` を採用する理由が整理されている
- `API`、`File`、`OData`、`SOAP`、`SFTP` 等の方式が明確
- `SAP - SAP` か `SAP - Non-SAP` かが明確
- `Cloud - On-Premise` 接続要否が整理されている

### 未決事項管理

- 不明点が未決事項として一覧化されている
- 業務側確認事項と技術側確認事項が分かれている
- 判断保留事項について期限と責任者がある

## 3.2 基本設計フェーズ

### 連携設計

- 1 連携 1 目的の原則が守られている
- `iFlow` 単位の責務が明確
- 正常系フローが整理されている
- 異常系フローが整理されている
- 再処理要否が整理されている

### 項目 / マッピング設計

- 入出力項目一覧がある
- 必須 / 任意が明確
- 型、桁数、フォーマットが明確
- マッピングルールが記載されている
- 固定値、変換、参照値が区別されている
- `Value Mapping` 利用要否が整理されている

### 運用観点

- 監視対象が整理されている
- 障害時の切り分け観点が整理されている
- 業務キー / 追跡キーが定義されている

## 3.3 詳細設計フェーズ

### iFlow 構成

- `iFlow` の開始、主処理、終了、例外処理が整理されている
- `Exception Subprocess` の扱いが定義されている
- 分岐条件が明確
- 外部接続箇所が明確
- ヘッダ / プロパティ設計が整理されている

### 実装方式

- 標準コンポーネント優先の方針になっている
- `Groovy Script` 使用箇所に理由がある
- スクリプト責務が明確
- ハードコード回避方針がある
- 認証・接続設定方針が整理されている

### 品質観点

- 調査しやすいログ設計になっている
- エラー時に必要な情報が取得できる設計になっている
- テスト観点を意識した設計になっている

## 3.4 開発フェーズ

### 実装品質

- 命名規約に従っている
- `iFlow` 構成が設計書と一致している
- 不要なステップがない
- スクリプトが過度に肥大化していない
- コメントや補足が必要最低限整理されている

### 接続・設定

- `Destination` 設定が正しい
- 認証設定が正しい
- 証明書設定が必要なら整理されている
- 接続先 URL / tenant / endpoint が確認されている

### エラー制御

- 異常時処理が実装されている
- 業務エラーとシステムエラーを区別できる
- 再送可否が判断できる
- 運用調査に必要な識別子が残る

## 3.5 開発レビュー / 内部レビュー

### AK 観点

- `iFlow` 責務分離は適切か
- 技術方式は妥当か
- スクリプト依存が過剰でないか
- 監視・調査性が担保されているか

### 田内観点

- 設計書との差分が整理されているか
- レビュー指摘が記録されているか
- 未解決事項が残ったまま次工程に進んでいないか
- レビュー完了条件が明確か

## 3.6 テスト準備フェーズ

### テスト計画

- テスト範囲が明確
- 単体 / 結合 / 異常系の観点がある
- テストデータが準備されている
- 外部接続先のテスト条件が確認されている

### ケース準備

- 正常系ケースがある
- 必須項目欠落ケースがある
- フォーマット異常ケースがある
- 接続失敗ケースがある
- 認証失敗ケースがある
- 再処理確認ケースがある

## 3.7 テスト実施フェーズ

### 実施管理

- テスト結果が記録されている
- 証跡が保存されている
- エラー内容が追跡可能
- 不具合票が管理されている

### 品質確認

- 設計どおりの結果になっている
- ログ / Trace / MPL で調査可能
- 不具合の再現条件が整理されている
- 改修後の再確認ができている

## 3.8 本番移行前フェーズ

### 移行準備

- 本番環境設定差分が整理されている
- 接続先設定が確認済み
- 認証情報 / 証明書準備が完了している
- 監視方法が定義されている
- 問い合わせ時の一次窓口が明確

### ドキュメント

- 設計書が最新版である
- テスト結果が揃っている
- 運用手順書がある
- 障害時対応方針がある

## 3.9 本番初期運用フェーズ

### 稼働確認

- 初回連携結果が確認されている
- 異常有無が監視されている
- 業務部門と結果整合が取れている

### 障害対応準備

- 一次切り分け手順が共有されている
- 再処理方法が整理されている
- エスカレーションルートが明確
- 問い合わせテンプレートがある

## 4. フェーズゲート

各フェーズ完了時の判断基準は以下。

### 要件整理完了

- 業務目的、対象システム、連携方式、未決事項が整理済み

### 基本設計完了

- 正常系 / 異常系、項目定義、監視観点が整理済み

### 詳細設計完了

- `iFlow` 構成、例外処理、接続方式、実装方針が整理済み

### 開発完了

- 実装レビュー完了
- 設計との差分整理済み

### テスト完了

- ケース実施済み
- 証跡あり
- 重要不具合クローズ済み

### 本番移行可

- 運用手順、監視、初期対応体制が整っている

## 5. 使用ルール

- 各案件で必要に応じて項目を追加してよい
- ただし最低限チェック項目は削除しない
- チェック結果は、完了 / 要対応 / 保留 などで管理する

## 6. 一言まとめ

`CPI` の品質は、実装の巧拙だけで決まらない。  
要件、設計、実装、テスト、監視、運用準備までを通して、各フェーズで確認を積み上げることが品質確保の本質である。
