# SAP Integration Suite L1 快速补课教程

来源：基于当前对客宣讲页左侧能力项整理  
整理日期：2026-03-13  
适用对象：需要在短时间内补齐 `SAP Integration Suite` 基础认知、并能在客户交流中应对基础问题的成员

## 这份教程怎么用

这不是深度开发手册，而是一份 `L1 速成版` 教程。目标只有两个：

- 让你听得懂客户在说什么
- 让你能对常见问题做出不露怯的基础回应

建议使用方式：

1. 先通读一遍全文，建立整体框架。
2. 重点记住每个模块的 `一句话定义`。
3. 重点背下每个模块的 `客户常问` 和 `推荐回答`。
4. 宣讲前只复习最后的 `一分钟速记版`。

## 一、先建立总框架

你这页里左侧讲的能力，本质上可以分成三层：

### 1. 基盘构建 / 接续设计

回答的是：`这个平台怎么搭起来，怎么跟系统连上。`

### 2. iFlow 设计 / 开发

回答的是：`具体的接口和数据流怎么做。`

### 3. 运营を見据えた設計

回答的是：`上线以后怎么稳、怎么查、怎么维护。`

如果客户问得比较泛，你就可以先用这三层来答：

`我们不仅做接口开发，也覆盖环境与连接准备，以及上线后的运维设计。`

## 二、基盘构建・接续设计

这一部分是平台落地的起点。你可以理解为：先把地基打好，再谈接口开发。

### 2.1 BTPサブアカウント設計

#### 一句话定义

`Subaccount` 是 SAP BTP 上承载应用、服务和权限配置的基础单元，相当于一个项目或环境的管理容器。

#### 为什么重要

如果 Subaccount 设计得乱，后面会出现：

- 环境边界不清
- 权限难管理
- 开发、测试、生产混在一起
- 后续运维风险大

#### L1 你要知道什么

- 一般会按环境拆分，比如开发、测试、生产。
- 不同项目或客户也可能拆不同 Subaccount。
- Integration Suite 的很多配置都是挂在这里面的。

#### 客户可能会问

`你们能支持环境规划吗？`

#### 推荐回答

`可以，我们在 Integration Suite 导入之前，会先考虑 BTP Subaccount 的环境划分、权限边界和后续运维方式，避免后面配置和管理变得复杂。`

### 2.2 Integration Suite 初期設定

#### 一句话定义

就是把 `SAP Integration Suite` 这个服务真正开通并完成最基本可用配置。

#### 为什么重要

买了服务不等于能直接做项目，还需要初始化平台能力。

#### L1 你要知道什么

- 包括服务启用
- 基础访问设置
- 相关能力的初始配置
- 与后续开发、连接、监控相关的基础准备

#### 客户可能会问

`你们做的是开发，还是连环境初始设置也能做？`

#### 推荐回答

`我们不仅做 iFlow 开发，也可以覆盖 Integration Suite 的初始配置，让项目从平台准备阶段就能顺利推进。`

### 2.3 Cloud Connector / On-Premise 接続

#### 一句话定义

`Cloud Connector` 是连接 SAP BTP 云平台与企业本地系统的重要桥梁。

#### 为什么重要

很多客户系统并不全在云上，尤其 S/4、本地 ERP、内网服务都可能在企业内部网络里。  
如果没有安全连接机制，云上 Integration Suite 就拿不到本地数据。

#### L1 你要知道什么

- 它解决的是 `云到本地` 的安全连接问题。
- 它不是简单“把网络打通”，而是要做受控、安全、可配置的连接。
- 很多混合集成项目都会涉及它。

#### 客户可能会问

`你们能处理 On-Premise 接続吗？`

#### 推荐回答

`可以，我们理解 Integration Suite 在混合环境下的连接前提，会结合 Cloud Connector 和 On-Premise 接続的方式来设计整体方案。`

### 2.4 認証・接続方式設計

#### 一句话定义

就是明确系统之间用什么方式安全通信、怎么互相信任、怎么做访问控制。

#### 常见关键词

- Basic Authentication
- OAuth
- Certificate
- Principal Propagation

#### L1 你要知道什么

- 认证解决的是“你是谁、能不能访问”。
- 接続方式解决的是“怎么连、怎么保证安全稳定”。
- 这是接口成功与否的前提，不是细枝末节。

#### 客户可能会问

`你们能处理 OAuth 和证书类配置吗？`

#### 推荐回答

`可以，这类内容属于 Integration 项目的基础能力之一，我们会在接続设计阶段就一起考虑，而不是等开发快结束才补。`

## 三、iFlow 设计・开发

这一部分是客户最容易直接联想到“Integration 工作”的部分。  
你可以把 `iFlow` 理解成：Integration Suite 里描述数据如何接收、处理、转换、发送的流程。

### 3.1 iFlow アーキテクチャ設計

#### 一句话定义

就是先想清楚接口流程怎么组织，而不是一上来直接拖图开发。

#### 为什么重要

一个能跑的 iFlow，不一定是一个好维护的 iFlow。  
设计阶段要考虑：

- 流程拆分是否合理
- 重试和异常处理怎么做
- 映射放在哪里
- 脚本是否必要
- 后续监控怎么查

#### L1 你要知道什么

- iFlow 不只是“画流程图”，它是有架构设计的。
- 好的 iFlow 设计会影响稳定性、可维护性和交接成本。

#### 客户可能会问

`你们是只会开发，还是也会做前面的设计？`

#### 推荐回答

`我们不是只做拖图开发，通常会先考虑 iFlow 的整体结构、异常路径、映射位置和后续运维性，再进入具体实现。`

### 3.2 API / File / OData / SOAP / REST 連携

#### 一句话定义

这是常见接口方式的集合，代表你们能处理不同类型系统之间的数据交换。

#### 你要快速理解这些词

- REST：现在最常见的 Web API 风格之一
- SOAP：较传统、规范更强的接口方式
- OData：SAP 场景中很常见的数据访问协议
- File：基于文件的集成方式
- API：广义上指系统对外暴露的调用接口

#### L1 你要知道什么

- 不同客户系统接口风格不同，没有一种方式能解决所有场景。
- SAP 项目里 OData 很常见。
- 老系统或 B2B 场景里文件接口也很常见。

#### 客户可能会问

`你们主要支持哪些接口方式？`

#### 推荐回答

`常见的 API、REST、SOAP、OData 和 File 类接口场景我们都能覆盖，具体会根据对端系统能力和业务要求选择合适方式。`

### 3.3 Value Mapping / Message Mapping

#### 一句话定义

- `Message Mapping`：把一种数据结构转换成另一种数据结构  
- `Value Mapping`：把一个值映射成另一个值

#### 举个最简单例子

- Message Mapping：把源系统的 XML 转成目标系统要求的 XML 或 JSON
- Value Mapping：把源系统中的 `A` 转成目标系统中的 `01`

#### 为什么重要

系统之间字段、结构、代码值往往不一致。  
映射能力几乎是集成项目的核心工作之一。

#### L1 你要知道什么

- Message Mapping 更偏结构转换
- Value Mapping 更偏代码值转换
- 这两类设计直接影响接口正确性

#### 客户可能会问

`复杂映射你们能处理吗？`

#### 推荐回答

`可以，映射是 Integration 项目里的基础核心能力之一，我们通常会根据结构复杂度选择合适的 Mapping 设计方式。`

### 3.4 Groovy Script 開発

#### 一句话定义

当标准图形配置不够用时，可以通过 `Groovy Script` 做更灵活的数据处理或逻辑控制。

#### 为什么重要

不是所有需求都能只靠拖图完成。  
例如：

- 特殊字段处理
- 条件判断
- 自定义格式转换
- 某些复杂逻辑控制

#### L1 你要知道什么

- Groovy 不是每个场景都必须用。
- 能不用脚本时，通常优先考虑标准方式，方便维护。
- 但复杂场景下，Groovy 很重要。

#### 客户可能会问

`你们做 CPI 时会涉及脚本开发吗？`

#### 推荐回答

`会，但我们通常会先评估是否能用标准方式实现；在确实有必要时，再用 Groovy Script 处理更复杂的逻辑。`

## 四、运营を見据えた設計

这部分很重要，因为它能把你和“只会做接口”的团队区分开。  
你要给客户传递的信号是：`我们不仅让它跑起来，也考虑它上线后怎么稳。`

### 4.1 例外処理設計

#### 一句话定义

就是提前设计：当接口出错、数据异常、外部系统失败时，系统应该怎么处理。

#### 为什么重要

真实项目里，异常不是“会不会发生”，而是“一定会发生”。

#### L1 你要知道什么

- 不能只设计成功路径
- 失败后的动作也要设计
- 否则上线后问题会集中爆发

#### 客户可能会问

`你们会考虑异常处理吗？`

#### 推荐回答

`会，我们在设计阶段就会考虑异常路径，而不是只关注正常传输成功的场景。`

### 4.2 エラーハンドリング設計

#### 一句话定义

就是明确：错误发生后如何识别、记录、通知、重试或人工介入。

#### 和例外处理的区别

- 例外处理更偏“异常场景设计”
- Error Handling 更偏“错误发生后的处理机制”

#### L1 你要知道什么

- 不是所有错误都应该自动重试
- 有些错误要报警
- 有些错误要人工处理

#### 客户可能会问

`如果接口失败了，你们一般怎么处理？`

#### 推荐回答

`我们会根据错误类型设计对应处理方式，比如记录日志、告警、重试或人工介入，避免所有错误都按同一种方式处理。`

### 4.3 監視観点を踏まえた設計

#### 一句话定义

在设计阶段就考虑，后面运维人员如何看监控、查问题、定位故障。

#### 为什么重要

很多接口开发时能跑，但一出问题就查不到原因。  
这通常不是运维能力差，而是设计时没考虑监控可视性。

#### L1 你要知道什么

- 好设计应该方便后续定位问题
- 日志、状态、关键节点要能看见
- 这会直接影响运维效率

#### 客户可能会问

`你们会把监控要求放到设计里吗？`

#### 推荐回答

`会，我们会在设计阶段就考虑后续的监控和问题定位方式，避免上线后出现‘能跑但不好查’的情况。`

### 4.4 保守性を考慮した設計

#### 一句话定义

就是让后续维护、变更、交接都更容易，而不是把接口做成只有原开发者看得懂。

#### 为什么重要

项目交付后，最常见的问题不是“今天能不能跑”，而是：

- 明年还能不能改
- 新人能不能接手
- 需求变化时会不会牵一发动全身

#### L1 你要知道什么

- 维护性是设计质量的一部分
- 命名、流程拆分、注释、脚本控制、映射方式都会影响维护性

#### 客户可能会问

`你们怎么保证后续可维护性？`

#### 推荐回答

`我们会在设计阶段就考虑后续维护，比如流程拆分、命名规范、错误处理和实现方式，尽量避免形成只有个别人能维护的接口。`

## 五、客户最容易问的 10 个基础问题

### 1. 你们做的是开发，还是也能做前面的环境准备？

`我们不仅做 iFlow 开发，也覆盖 BTP 环境准备、Integration Suite 初始配置、连接设计等前置工作。`

### 2. 你们能处理本地系统连接吗？

`可以，像 Cloud Connector 和 On-Premise 接続这类混合集成前提能力，我们是会一并考虑的。`

### 3. iFlow 是什么？

`可以理解为 Integration Suite 里定义接口流程的核心对象，它描述数据如何接收、转换、处理和发送。`

### 4. Mapping 是什么？

`就是把源系统的数据结构或代码值，转换成目标系统需要的结构和含义。`

### 5. Groovy Script 是必须的吗？

`不一定。标准方式能实现时通常优先标准方式，复杂场景下才会使用脚本增强灵活性。`

### 6. 你们只负责把接口做通吗？

`不是，我们也会考虑异常处理、错误应对、监控和后续维护性。`

### 7. 你们做 API 类接口吗？

`可以，常见的 API、REST、SOAP、OData、File 等接口场景都在我们的覆盖范围内。`

### 8. 认证和证书这些你们管吗？

`会，这些是接口成功落地的基础条件，通常需要在接续设计阶段一起考虑。`

### 9. 你们怎么保证上线后好维护？

`我们会在设计阶段就考虑可维护性，包括流程结构、命名、错误处理和监控可视性。`

### 10. 你们和只会做 CPI 开发的团队有什么区别？

`我们的覆盖范围不只是开发本身，还包括前期环境和连接准备，以及上线后的运维设计。`

## 六、你对客户时最安全的表达方式

如果你不确定细节，不要硬讲技术深水区。  
你可以用下面这种表达，既稳，又不容易露怯。

### 通用回答模板

`这部分我们是有对应经验的。通常会先结合客户现有环境和接口要求，确认连接方式、设计方式和后续运维要求，再选择合适的实现方案。`

### 遇到细节问题的缓冲说法

`这个点我先从整体设计思路回答您，如果需要，我们后续也可以把具体实现方式再展开说明。`

### 避免踩坑

不要轻易说：

- `这个很简单`
- `这个肯定没问题`
- `这个我们都能做`

更稳妥的说法是：

- `这是我们覆盖范围内的典型场景`
- `这类需求我们通常会结合现有环境来设计`
- `这个方向我们可以支持，具体方式会看系统边界和业务要求`

## 八、建议你宣讲前怎么准备

### 宣讲前 30 分钟

- 先背熟“三层能力”
- 再背熟“10 个基础问题”
- 最后把“一句总话术”和“一句差异化话术”念 3 遍

### 宣讲时的原则

- 先讲整体，再讲细节
- 先讲客户价值，再讲技术术语
- 先讲设计思路，再讲实现方式

### 如果客户追问很深

你可以先稳住节奏：

`这个点我先从项目设计和落地方式上回答您，具体到实现细节，我们也可以后续再进一步展开。`
