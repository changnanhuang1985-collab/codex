# 3/17向け Integration 訴求方針とチーム能力構築計画

## 1. 前提

本資料は、`大黄` が統括し、`山田`、`shishi`、`AK`、`wl`、`tu`、`yu`、`田内` の観点を踏まえて整理した、`3/17` 顧客会議向けの `BTP Integration` 訴求方針と、今後のチーム能力構築計画である。

今回の前提は明確である。

- 今回は `CAP` や `BTP Extension` 全般ではなく、`Integration` に集中して訴求する
- 前回、`山田` から「3名に経験がある」と説明しているため、今回それを否定するような説明は避ける
- ただし、`CPI` を過大に強く見せる表現も避ける
- 顧客にとって重要なのは、今どこまで安全に任せられるかである

## 2. 結論

### 2.1 3/17での基本メッセージ

`当チームでは、3名の経験者を中心に SAP Integration Suite / CPI の体制を整えており、現在は設計・開発・テスト・運用の標準化を進めながら、まずは切り出しやすい Integration スコープから高品質にご支援できる体制を作っています。`

これが今回の中心メッセージである。

### 2.2 伝え方の基本方針

- `経験者が3名いる` は維持する
- ただし `経験豊富なCPI専門集団` とは言わない
- `個人の経験` と `チームとしての体制化` を分けて説明する
- `今すぐ何でも対応可能` ではなく、`まずは任せやすい範囲から堅実に対応可能` と見せる

## 3. 3/17での核心Point

今回の核心Pointは 4 つに絞るべきである。

## Point 1. 3名の経験者を核にした Integration 体制

言いたいこと：

- `CPI / Integration` に関連する設計・開発・支援経験を持つメンバーが 3 名いる
- その 3 名を核としてチーム体制を構築している

狙い：

- 前回説明との整合を保つ
- “ゼロからこれから勉強するチーム” という印象を避ける

## Point 2. 過大な実績ではなく、再現可能な delivery model を構築中

言いたいこと：

- 個人経験をチームとして再現可能な形にするため、設計標準・開発標準・テスト標準・運用標準を整備している

狙い：

- 実績が薄い部分を“仕組み化”で補っていると見せる
- 顧客に安心感を与える

## Point 3. まずは切り出しやすい Integration スコープから対応可能

言いたいこと：

- 最初から大規模統合全体ではなく、`iFlow`、API 連携、OData、File、SFTP、基本マッピング、試験・運用設計など、切り出しやすい領域から高品質に支援可能

狙い：

- 顧客に「まず一部を任せてみよう」と思ってもらう

## Point 4. 技術だけでなく、対日品質での交付を重視

言いたいこと：

- 日本語設計書
- レビュー対応
- 試験証跡
- 運用観点

まで含めて考えている

狙い：

- 価格だけでなく、安心して使える offshore team として見せる

## 4. 3/17での推奨説明順序

以下の順番で説明するのが最も自然である。

### 1. チームの位置づけ

- 当チームは新しいが、主要メンバーは大手出身で、大型案件経験を持つ

### 2. Integration 体制

- 3名の経験者を核に、`SAP Integration Suite / CPI` を重点領域として体制化している

### 3. 今の強み

- `iFlow` 設計・開発
- マッピング
- SAP / Non-SAP 連携
- API / OData / File / SFTP
- テスト・運用設計

### 4. 今の進め方

- 標準化を進めながら、まずは切り出しやすい範囲から支援する

### 5. 協業イメージ

- 日本側チームのもとで、設計・開発・試験・運用初期支援を堅実に担当する

## 5. 3/17でそのまま使える口頭説明

`Integration領域については、現在3名の経験者を中心に、SAP Integration Suite / CPI の体制を整えています。メンバー個人としては、iFlow 設計、マッピング、SAP / Non-SAP 連携、接続設定、試験・運用観点まで、関連する設計・開発・支援経験を有しています。`

`一方で、チームとしては今まさにこの領域を重点強化している段階でもあるため、過大な実績表現ではなく、設計標準・開発標準・テスト・運用標準を整備しながら、まずは切り出しやすい Integration スコープから高品質にご支援できる体制を作っています。`

`最初から大きな範囲をお任せいただくというより、まずは任せやすい範囲で着実に価値をお返しし、その中で継続的に強化していきたいと考えています。`

## 6. 山田から見た外向け表現の注意点

`山田` 観点での結論は以下。

- “強い” より “安心して任せられる” を優先する
- “豊富な実績” より “3名の経験者を核に体制化している” の方が信頼される
- “これから強化する” だけでは弱いので、“すでに標準化を始めている” を必ず入れる
- 顧客に最初の一歩を踏ませるため、“まずは切り出しやすい範囲” という逃げ道を作る

## 8. チーム能力構築計画

大黄としては、能力構築は以下の 3 段階で進めるべきと判断する。

## Phase 1. 最小交付能力の確立

目的：

- `CPI` の標準的な小中規模案件を安全に回せる状態にする

優先項目：

- `iFlow` 設計標準
- `iFlow` 開発標準
- 命名規約
- マッピング定義テンプレート
- テストケーステンプレート
- 運用 / 障害切り分けテンプレート

対象範囲：

- API / OData / File / SFTP
- 基本マッピング
- エラー処理
- 監視、ログ、再処理

## Phase 2. 提案力と実績化

目的：

- 技術力を顧客が理解できる形に変える

優先项：

- 匿名化ケース事例
- PPT 能力紹介
- 役割分担説明
- “まず任せられるスコープ” の明文化

対象範囲：

- `SAP - SAP`
- `SAP - Non-SAP`
- `Cloud - On-Premise`
- 小中規模 Integration スコープ

## Phase 3. 中期強化

目的：

- `CPI` をチームの一つの主戦力に育てる

優先项：

- 複雑マッピング
- 例外制御高度化
- API Management 連携
- 運用高度化
- 案件別ナレッジ蓄積

必要に応じて将来検討：

- Event 系
- B2B 系
- Governance 系

## 10. 最終結論

今回、3/17 で一番大事なのは、

`我々はもう十分強い`

と見せることではない。

正しくは、

`3名の経験者を核に、CPI を中心とした Integration 体制を立ち上げており、標準化と品質管理を進めながら、まずは任せやすい範囲から堅実に支援できる`

と見せることである。

これが、前回説明との整合を保ちつつ、信頼を落とさず、次の案件につなげる最も現実的な伝え方である。
