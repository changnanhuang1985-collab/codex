# 3/17向け BTP Integration 能力訴求・立上げ戦略

## 1. 背景

現在、当チームは `2025年後半` に立ち上がった比較的新しい組織であり、`BTP Integration` 領域ではまだ市場に対して十分な実績蓄積を見せられる段階ではない。

一方で、以下のような強い事実もある。

- 主要メンバーは `Accenture`、`IBM` 等の大手出身
- `2000+ 人月` 規模の大型案件経験を持つメンバーが在籍
- `CAP` を中心とした `BTP` 拡張開発能力は強い
- 前任チームが失敗した案件を立て直し、社内評価を大きく改善した実績がある
- 現在進行中の案件を通じて、チームとしての実行力と信頼回復力は示しつつある

ただし、現実として以下も認識すべきである。

- `CPI` はチームの相対的弱点であり、現時点で過大な表現は危険
- 顧客は「BTP の何でも屋」ではなく、「任せても大丈夫な実行チーム」を求めている

この前提を踏まえ、本戦略では次の 2 つを同時に進める。

1. `3/17` までに、顧客に伝わる形で `BTP Integration` 能力を訴求する
2. 短期間で `CPI` 中心の最小実行能力と標準化を整える

## 2. 大黄としての結論

### 2.1 第一の判断

今、顧客に対してやるべきことは、「CPI が非常に強いチームです」と言い切ることではない。

正しくは、

`当社は BTP 全体の拡張開発力を基盤に、Integration 領域を次の重点テーマとして実行体制化しており、CPI を中心とした設計・開発・標準化・提案準備を短期間で立ち上げている`

と伝えるべきである。

### 2.2 第二の判断

顧客が最初に見ているのは「現時点の絶対実績量」ではなく、

- 本人たちがどれだけ現実を分かっているか
- どこまでなら安全に任せられるか
- 弱いところをどう補強しているか
- 再現可能な進め方を持っているか

である。

したがって、今回の勝ち筋は以下である。

- `CAP` を含む `BTP` 総合力を土台として示す
- `Integration` は今後の重点強化領域であると明確にする
- `CPI` は「過大な実績」ではなく「立上げ済みの実行能力」として見せる
- 標準化、設計書、レビュー、テスト、運用設計を先に見せて信頼を取る

## 3. 顧客にどう見せるべきか

## 3.1 訴求の軸

顧客への訴求軸は、次の順番にする。

1. `BTP` チームとしての母体の強さ
2. 実行力の証拠
3. `Integration` を重点強化していること
4. `CPI` を安全に進める標準化・品質管理の仕組み
5. まず任せやすいスコープ

## 3.2 伝えるべきストーリー

推奨ストーリーは以下。

### ステップ 1

当チームは新しい組織だが、メンバー個人は大手出身で、大規模案件経験が豊富である。

### ステップ 2

単なる人材寄せ集めではなく、難しい案件の立て直しを通じて、実行力と信頼回復力を証明している。

### ステップ 3

現在の主戦場は `CAP` を中心とした `BTP` 拡張だが、顧客需要を踏まえ `Integration` を重点強化領域として立ち上げている。

### ステップ 4

そのために、`CPI` 開発だけでなく、設計標準、開発標準、テスト、運用、提案資料まで短期間で標準化している。

### ステップ 5

最初から何でも請けるのではなく、まずは `CPI` を中心とした切り出しやすいスコープから、堅実に価値を出す。

## 4. 顧客向けに使うべき表現

## 4.1 使うべき言い方

- 当社は `BTP` 全体の開発基盤を持つ中で、`Integration` を重点テーマとして強化しています
- `CAP` を中心とした拡張開発力に加え、`CPI` を中心とする連携設計・実装・標準化を進めています
- 実績の誇張ではなく、設計・標準化・品質管理を先に整えることで、安心して任せられる `Integration` チームを作っています
- 小さく切り出しやすい `CPI` 領域から、着実にご支援可能です

## 4.2 避けるべき言い方

- `CPI はかなり得意です`
- `Integration 実績が豊富です`
- `何でもすぐできます`
- `BTP は全部任せてください`

理由：

- 現時点のチーム実態とズレる
- あとで深掘りされたときに弱い
- 日本の顧客には誠実さと再現性の方が刺さる

## 5. 今回の訴求ポイント

## 5.1 チームの土台の強さ

- 大手コンサル / IT 企業出身メンバーで構成
- 大型案件の設計・開発・導入経験あり
- 個人の経験値は高い

## 5.2 組織としての実行力

- 立ち上がったばかりの新チームでありながら、難航案件の立て直しに成功
- 「失敗案件を安定化させた」という事実は、顧客に対する非常に強い信頼材料

## 5.3 BTP 拡張に強い

- 現時点での主力は `CAP`
- `BTP Extension` を深く理解している
- そのため、`Integration` も単独機能としてではなく、`BTP` 全体像の中で設計できる

## 5.4 Integration は重点投資中

- `CPI` を中心に、連携標準、設計標準、運用標準を整備中
- 単なる開発者育成ではなく、チームとしての再現可能な delivery model を作っている

## 7. どこまでを今の実力として言うか

## 7.1 今、言ってよいこと

- `CPI` を中心にした `Integration` 能力構築を重点的に進めている
- 設計・開発・標準化・品質管理の枠組みを整えている
- `BTP Extension` と組み合わせた連携提案ができる

## 7.2 まだ言い過ぎなこと

- 大規模 `Integration` プログラムを単独主導できる

## 8. まず受けに行くべきスコープ

最初の受注では、以下のような `CPI` スコープを狙うべき。

- `SAP - SAP` の標準寄り連携
- `SAP - Non-SAP` の API / File / SFTP / OData 連携
- `iFlow` 設計・開発
- マッピング定義
- テスト支援
- 運用設計
- 小規模の連携群

逆に、最初から避けた方がよいもの：

- 高難度の大規模統合全体統括
- 実績のない高度な `Event` / `B2B` 案件を主戦場として名乗ること
- 顧客前面で `Integration Strategy` 全体を一社で引っ張る立場を過剰に取りに行くこと

## 10. BTP Integration 能力を短期で立ち上げる方針

## 10.1 目標設定

短期目標は「CPI の深い専門集団になること」ではない。

まず目指すべきは、

`標準的な CPI 案件を、設計・実装・試験・運用まで一貫して安全に回せる最小チーム`

である。

## 10.2 立上げの 4 本柱

### 1. 標準化

- 設計標準
- 開発標準
- 命名規約
- テスト標準
- 運用標準

### 2. スコープ定義

- できること
- まだやらないこと
- 支援可能な案件タイプ

### 3. テンプレート化

- IF 設計書
- マッピング表
- テストケース
- 障害切り分けシート

### 4. 事例化

- 実案件
- 匿名化ケース
- デモ構成

## 10.3 能力育成の優先順位

1. `Cloud Integration` 基本構成
2. `iFlow` 設計・実装
3. `Message Mapping / Value Mapping`
4. `OData / API / SFTP / File`
5. 監視、エラー、再処理
6. 顧客向け説明力

## 13. 追加論点：前回説明との整合の取り方

### 13.1 現在の懸念

前回の試講会では、`山田` から顧客に対して「3 名に開発経験がある」と説明している。

このため、今回の正式会議で急に

- `CPI はまだ弱い`
- `あまりやったことがない`
- `ほぼ未経験`

のように聞こえる説明をすると、顧客は

- 前回と話が違う
- 実態を曖昧に話している
- チームの自己認識がぶれている

と受け取る可能性がある。

### 13.2 大黄としての判断

ここでやるべきことは、前回発言を否定することではない。

正しくは、前回の「3 名に経験がある」を以下のように定義し直して説明することである。

- `Integration / CPI に関する開発・設計・実装・支援経験を持つメンバーが 3 名いる`
- ただし、チーム全体としては現在 `CPI` を重点強化中であり、標準化と delivery model を整えている

つまり、

- 個人経験はある
- ただしチームとしては今まさに再現可能な形にしている最中

という整理にする。

これなら、

- 前回発言との矛盾を避けられる
- 誇張にもならない
- 現在の立上げ状況も自然に説明できる

## 14. 3/17での集成能力の紹介方針

## 14.1 絶対に避けるべき言い方

- `CPI はまだほとんどできません`
- `実績はほぼありません`
- `これから勉強します`

これらは事実の一部を誇張して弱く見せすぎており、前回説明とも整合しない。

## 14.2 推奨する基本表現

推奨する言い方は以下。

`当チームでは、Integration 領域について 3 名の経験者を中心に、SAP Integration Suite / CPI を重点強化しています。個人ベースでは設計・開発・支援経験を有しており、現在はそれをチームとして再現可能な delivery model にするため、設計標準、開発標準、テスト・運用標準の整備を進めています。`

この表現の良い点は以下。

- 前回の「3 名経験者」と矛盾しない
- 実績を盛りすぎない
- 今まさに強化中であることを前向きに言える
- 顧客に「今後任せられそう」と思わせやすい

## 14.3 顧客への見せ方の軸

3/17 では、Integration 能力を次の 3 層で説明するのがよい。

### 第 1 層：人の経験

- 3 名の経験者がいる
- メンバーは大手出身で大型案件経験がある
- `SAP - SAP`、`SAP - Non-SAP`、API、File、OData 等の理解がある

### 第 2 層：チームとしての実行体制

- 個人経験をチーム delivery に変えるための標準化を進めている
- 設計、実装、試験、運用まで一貫で考えている
- 小中規模の切り出しスコープなら安全に担当できる

### 第 3 層：今後の伸ばし方

- `CPI` を重点強化領域として位置付けている
- `CAP` を含む BTP 全体理解を土台に、Integration を拡張していく
- 顧客と一緒に案件を通じて強化していける
