# 日本向け BTP パートナー営業アピール整理
## 対象：CTC 類似の SAP BTP / Integration 推進チーム向け

## 1. 本資料の目的

本資料は、日本のお客様、特に `CTC` に類する `SAP BTP` 推進チームに対して、当社がどのような強みを、どの順番で、どのような言葉で説明すれば「一緒に進められそうだ」と感じていただけるかを整理した営業・提案支援資料である。

想定する案件規模は `30人月前後` の中規模案件であり、最初の受注を目的とした現実的なアピールポイントに絞っている。

## 2. pm / ss / jk から見た前提整理

### pm 観点

- 30人月規模の案件では、「安い」だけでは受注できない
- お客様は、納期、品質、説明責任、体制安定性を重視する
- 最初の案件では「全部任せられる会社」よりも、「任せても事故が起きにくい会社」であることの方が重要

### ss 観点

- 日本の `BTP` 推進チームは、単なる実装リソースではなく、「顧客説明に耐える技術パートナー」を探している
- 特に `CTC` 類似のチームは、自社の顧客に再提案・再説明できる相手を求める
- そのため、営業では「技術力」だけでなく、「説明可能性」「整理力」「日本向け品質」が刺さる

### jk 観点

- `BTP` 案件では、アーキテクチャの考え方が浅い会社はすぐ見抜かれる
- 逆に、大規模実績が少なくても、責務分離、標準優先、API / Integration / Event の使い分けをきちんと説明できれば信頼されやすい
- 最初の受注では、“全部できる”より “どこまでを高品質に担当できるか” を明確にする方が強い

## 3. この顧客が本当に見ているポイント

日本の `CTC` 類似チームが外部パートナーを見る際、実際には以下を確認していることが多い。

### 3.1 日本向けコミュニケーションに耐えるか

- 日本語で設計内容を説明できるか
- 指摘に対して論点を整理して返せるか
- 曖昧な状態で実装を始めないか
- 変更時に影響範囲をきちんと示せるか

### 3.2 BTP を本当に理解しているか

- `Cloud Integration` を単なる IF 開発ツールとしてしか見ていないか
- `API Management`、`Event Mesh`、`BTP` 拡張の役割を説明できるか
- `PI/PO` から `Integration Suite` への考え方を現実的に話せるか

### 3.3 小規模案件でも安心して任せられるか

- 30人月前後では、体制過大も体制不足も不安要素になる
- 少人数でも、役割、レビュー、品質管理が整っているかを見ている

### 3.4 顧客前面に出しても危なくないか

- 提案先のお客様にそのまま連れていけるか
- 顧客との定例やレビューで、発言の質が保てるか
- 技術だけでなく、期待値調整もできるか

## 4. まずアピールすべき能力

最初に押し出すべきは、以下の 5 点である。

## 4.1 日本向け品質での設計・実装能力

言いたいこと：
- 日本案件で求められる設計粒度、レビュー対応、証跡整備に対応可能

伝え方の例：
- 日本語での設計書、QA 管理、指摘反映、変更影響整理まで含めて対応可能です
- 実装だけでなく、レビューに耐える説明資料と試験証跡の整備も重視しています

刺さる理由：
- `CTC` 類似チームは、下流作業だけでなく、顧客説明で困らない相手を求めているため

## 4.2 BTP / Integration の現実的なアーキテクチャ理解

言いたいこと：
- SAP 標準を尊重しつつ、BTP 上でどこを拡張し、どこを標準活用するべきかを判断できる

伝え方の例：
- `Cloud Integration`、`API Management`、`BTP` 拡張を役割分担して設計します
- `PI/PO` の全面即時更改ではなく、段階移行を前提とした現実的な方式整理が可能です

刺さる理由：
- “BTP を知っているだけ” ではなく、“お客様環境でどう使うか” が語れることが重要なため

## 4.3 少人数高密度で回せる実行体制

言いたいこと：
- 大人数ではないが、30人月前後の案件であれば、必要十分な役割分担と品質管理で対応できる

伝え方の例：
- 5 名前後の体制で、コンサル、アーキテクト、開発リード、開発・試験を分担し、密度高く進めます
- うち 3 名は対日案件経験があり、レビュー・設計・顧客説明を支えます

刺さる理由：
- 日本側は「人をたくさん出せる会社」より、「事故らない進め方ができる会社」を求める場合が多いため

## 4.4 設計から試験・ハイパーケアまで切れ目なく支援できること

言いたいこと：
- 要件整理の補助から、設計、開発、試験、運用初期支援まで一貫対応できる

伝え方の例：
- IF 設計だけで終わらず、テスト観点整理、不具合解析、ハイパーケアまで支援可能です
- そのため、日本側チームの手離れを良くできます

刺さる理由：
- BTP 案件では、実装後の障害切り分けまで含めて負荷が高いため

## 4.5 説明できる実績の持ち方

言いたいこと：
- 過大な実績表現を避け、責任範囲を明確にした形で事例説明できる

伝え方の例：
- 大規模案件全体を自社実績として語るのではなく、担当範囲を明確にしてご説明します
- 設計、実装、試験、運用支援のどこを担ったかを明確にします

刺さる理由：
- 日本の SI チームは、実績の盛り方に敏感であり、誠実な説明が信頼につながるため

## 5. 顧客にどう紹介するか

## 5.1 先に言うべきこと

商談の冒頭では、以下の順で話すのがよい。

1. 当社は `SAP BTP / Integration` 領域に絞って支援できる
2. 日本向け品質を意識した設計・実装・試験対応が可能
3. 30人月前後の案件で、少人数高密度型の支援に適している
4. `PI/PO` 更改や `BTP` 活用の初期案件で、無理のない移行・設計支援ができる

## 5.2 途中で補強すべきこと

- 類似ケースを匿名事例として説明する
- 体制と役割を具体的に示す
- “できること” だけでなく “やらないこと / 無理に広げないこと” も示す

## 5.3 最後に伝えるべきこと

- 最初の 1 件を安心して任せてもらえるよう、責任範囲を明確にして進める
- 日本側の負荷を増やさず、むしろ整理・設計・証跡面で支援できる
- 小さく始めて、継続案件につなげることができる

## 6. 顧客向けメッセージ例

### パターン 1：堅実型

当社は、`SAP BTP` と `Integration Suite` を活用した設計・開発支援に特化し、対日案件で求められる設計品質、レビュー対応、試験証跡整備まで含めて支援可能です。大規模体制を前提とせず、30人月前後の案件に対して、少人数でも役割分担を明確にした高密度型の支援をご提供できます。

### パターン 2：移行支援型

`PI/PO` をご利用中のお客様に対しては、全面更改を前提にした非現実的な議論ではなく、新規投資をどこから `BTP / Integration Suite` に寄せるべきかという現実的な整理からご支援できます。標準を活かしつつ、必要な拡張だけを適切に実装する方針を重視しています。

### パターン 3：パートナー協業型

当社は、元請様や日本側 `BTP` 推進チームの後ろで、設計整理、実装、試験、ハイパーケアを安定的に支える協業型パートナーとして価値を発揮できます。顧客前面に出す説明にも耐えうる日本語ドキュメントと論点整理を重視しています。

## 7. 絶対に避けたいアピール

以下は逆効果になりやすいため避けるべきである。

- 「何でもできます」
- 「大規模案件も全部対応できます」
- 「人数はすぐ増やせます」
- 「BTP は全部わかります」
- 「安いので使ってください」

理由：
- 日本の `BTP` チームは、営業トークよりも再現性と誠実さを重視するため
- 強すぎる表現は、逆にレビュー不安や炎上リスクを想起させるため

## 8. 30人月案件を取りに行くための勝ち筋

### 8.1 最初の受注は“全部取り”を狙わない

狙うべき領域例：
- IF 設計 + `iFlow` 開発
- `BTP` 拡張の一部領域
- 結合試験 / ハイパーケア支援
- `PI/PO` 更改影響整理と新規 IF の `Integration Suite` 化

### 8.2 “安心して切り出せる単位” を示す

顧客が任せやすい単位の例：
- 5 - 8 本程度の IF 群
- 1 業務領域単位の API / Integration 対応
- 設計から試験までの一連対応

### 8.3 受注前から進め方を見せる

提案時に見せるとよいもの：
- 役割分担表
- 設計・レビューの進め方
- 品質ゲート
- 課題管理・変更管理の方法
- 日本語ドキュメントサンプル

## 9. ss と jk から見た商談ストーリー

### ss の整理

顧客に刺さるストーリーは以下である。

- お客様は `BTP` を使いたいのではなく、顧客案件を安全に進めたい
- 当社は、BTP を理解したうえで、日本向けに安全に進める方法を持っている
- そのため、最初の 30人月規模案件で一緒に進める相手として適している

### jk の整理

技術観点で信頼されるには以下が必要である。

- `Cloud Integration`、`API Management`、`BTP` 拡張の役割を整理して語る
- 標準優先、責務分離、監視前提の設計思想を示す
- `PI/PO` をただ否定せず、段階移行の考え方を話す

## 10. 実際に使える営業トークの骨子

まずは、すべてを任せていただくというより、`SAP BTP / Integration` 領域で切り出しやすい範囲からご一緒したいと考えています。当社は少人数体制ですが、対日案件経験のあるメンバーを中心に、設計、開発、試験、ハイパーケアまで一貫して支援できます。特に、`PI/PO` 更改や `BTP` 活用の初期案件において、標準を尊重しながら無理のないアーキテクチャと進め方をご提案できる点が強みです。

## 11. 次に準備すべきもの

この内容を実際の営業に使うためには、次の 3 点を合わせて準備すると効果が高い。

1. 匿名化ケース事例
2. 役割別体制説明
3. 日本語の設計書 / 試験書サンプル

本フォルダ以外で既にある資料と組み合わせるなら、以下が使いやすい。

- ケース事例資料
- 標準・規約ドキュメント
- 能力マトリクス

## 12. まとめ

この顧客に対して最も重要なのは、「技術力がある会社」に見せることではなく、「日本向け `BTP` 案件を、無理なく、誠実に、品質高く一緒に進められる会社」に見せることである。

そのため、営業では次の 3 点に絞って伝えるべきである。

- 日本向け品質での設計・実装・試験対応力
- `BTP / Integration` の現実的なアーキテクチャ理解
- 30人月前後を安心して任せられる少人数高密度体制
