# 例外処理および監視設計

## 目的

CPI ベースの Ariba-S/4HANA 連携における例外分類、監視方式、運用初動の前提を整理する。

## 例外分類

### 区分 A: 業務バリデーションエラー

- 必須項目欠落
- コード変換不可
- サプライヤ未登録などの参照不整合
- 対応方針: 処理停止、業務的に理解可能なエラー文言を残し、修正後再処理可能とする

### 区分 B: 技術接続エラー

- Endpoint timeout
- 認証失敗
- HTTP 5xx
- 対応方針: 技術原因を記録し、再試行可否とエスカレーション判断を行う

### 区分 C: 想定外内部エラー

- Groovy 実行時エラー
- Payload 解析失敗
- 内部処理例外
- 対応方針: Exception Subprocess へ遷移し、通知および調査用情報を保持する

## 監視方針

- 標準監視は MPL を利用する
- Trace は不具合解析時または計画的検証時のみ有効化する
- 業務キーで検索可能なログ設計とする
- 通知には IF ID、業務キー、発生時刻、概要原因を含める

## 標準ログ項目

- IF ID
- シナリオ名
- 送信元システム
- 受信先システム
- 業務キー
- Correlation ID
- 処理ステータス
- エラー区分
- エラー概要

## 通知方針

- 本番重大障害: 即時通知
- 非本番障害: 業務時間内通知
- 同一原因の多発障害: 可能な限り集約通知

## 再処理方針

- 業務修正可能または一時的技術障害のみ対象とする
- 再処理時は実施者、時刻、理由を記録する
- 重複影響を確認したうえで再送する

## レビューポイント

- 障害発生箇所を実務的な時間で特定できるか
- ログ項目が業務 / 技術双方の調査に足りているか
- 通知先とエスカレーション経路が定義されているか
