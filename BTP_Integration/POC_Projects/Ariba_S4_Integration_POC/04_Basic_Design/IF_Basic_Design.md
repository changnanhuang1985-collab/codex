# IF 基本設計書

## 文書目的

本書は、SAP Integration Suite 上で実装する Ariba-S/4HANA 連携インターフェースの基本設計方針を定義するものである。

## 対象範囲

- サプライヤマスタ連携
- 購買トランザクション連携
- 共通エラー通知および監視設計

## 設計方針

- 1 つの業務シナリオに対して 1 つの主 iFlow を原則とする
- 共通例外処理は再利用可能な形で分離する
- メッセージ変換は明示的に設計し、暗黙ロジックを避ける
- 業務キーは E2E で保持し、調査可能性を確保する
- 再処理前提の有無を設計時点で明確にする

## IF 一覧

| IF ID | IF 名称 | 方向 | トリガー | 頻度 |
| --- | --- | --- | --- | --- |
| IF-AR-S4-001 | サプライヤマスタ連携 | Ariba -> S/4 | API / Event | 準リアルタイム |
| IF-AR-S4-002 | 購買トランザクション連携 | Ariba -> S/4 | API / Event | 準リアルタイム |
| IF-AR-S4-003 | 購買応答返却連携 | S/4 -> Ariba | API / Batch | 準リアルタイムまたは定期 |
| IF-AR-S4-004 | エラー通知連携 | CPI -> Mail / Alert | 例外発生時 | 即時 |

## 業務処理概要

### サプライヤマスタ連携

1. Ariba からサプライヤ関連 payload を受信する
2. CPI で必須項目およびメッセージ構造を検証する
3. S/4 向け canonical 形式へマッピングする
4. S/4 API または OData を呼び出す
5. 処理結果を業務キーおよび correlation ID とともに記録する

### 購買トランザクション連携

1. Ariba から購買トランザクションデータを受信する
2. CPI でメッセージ種別および業務前提を確認する
3. S/4 互換構造へ変換する
4. S/4 サービスを呼び出す
5. 必要に応じて応答結果を正規化し、ログへ記録する

## 入出力設計方針

- ソース固有項目は過度に共通化しない
- 共通業務項目は IF 間で整合したマッピングとする
- ターゲット必須項目はアダプタ呼出前に検証する
- コード変換は統制が必要なものに限り Value Mapping を利用する

## 業務キー方針

- Supplier ID
- Ariba Document ID
- Purchase Order Number
- MPL ID
- CPI 生成 Correlation ID

## 非機能設計方針

- 追跡性: 必須
- 復旧性: 業務的に修正可能なエラーは再処理可能
- 冪等性: マスタ同期系では考慮必須
- 監査性: 処理経路と結果が再構成可能であること

## レビューポイント

- 送信元 / 受信先の責務分担が明確か
- 必須項目がすべてマッピング対象または除外理由付きで整理されているか
- 正常系と異常系が同等の粒度で設計されているか
- 再送 / 重複制御の前提が記載されているか
