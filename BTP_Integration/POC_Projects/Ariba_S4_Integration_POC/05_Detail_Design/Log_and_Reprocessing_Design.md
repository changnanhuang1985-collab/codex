# ログおよび再処理設計

## 目的

調査容易性を備えたログ設計と、統制された再処理方針を定義する。

## ログ設計方針

- 調査および監査に必要な情報のみを記録する
- 機微情報は承認なく平文出力しない
- 業務キーおよび correlation ID を一貫して保持する

## 必須ログ項目

- IF ID
- iFlow 名
- 送信元メッセージ ID
- CPI MPL ID
- 業務キー
- 処理開始時刻
- 処理終了時刻
- 最終ステータス
- エラー概要

## 再処理可否基準

### 再処理可

- 一時的な backend timeout
- 復旧後の認証トークン障害
- 業務データ修正後の再送

### 再処理不可

- 重複登録リスクが不明なケース
- payload 整合性が担保できないケース
- ロジック不具合が未改修のケース

## 再処理統制

- 再処理依頼にはチケット番号を紐付ける
- 重複影響確認を実施者が明示する
- 再処理結果は同一業務キーと新実行時刻で記録する

## 運用補足

本番環境での再処理は、承認フローおよび証跡保持を前提とする。
