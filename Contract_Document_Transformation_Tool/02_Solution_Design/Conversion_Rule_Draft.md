# Conversion Rule Draft

## 1. 目的

本文档是当前项目的第一版转换规则草案。

它不是最终可执行规则，而是让项目从“知道要做什么”进入“知道准备怎么做”。

## 2. 规则设计原则

- 真实需求 `input` 决定本次项目变量
- 历史 `output` 只作为模板和格式参考
- 不允许把历史项目特有内容直接照搬到当前需求
- 高风险字段必须进入 review

## 3. 首版规则分层

### 3.1 输入识别规则

- 根据文件名识别输入文档类型
- 根据文件名和正文识别客户名称、项目主题、编号
- 对无法仅靠文件名识别的信息，标记为待正文抽取

### 3.2 输出选择规则

- 根据当前需求目标，决定本次需要生成哪些内部文档类别
- 文书型、台账型、检查型文档分别走不同模板

### 3.3 模板应用规则

- 固定文字来自参考模板拆解结果
- 动态字段来自当前 `input`
- 不适用的历史项目字段不得直接继承

### 3.4 文件命名规则

建议首版采用：

`<期间或年度>_<内部文档类型>_<客户名称>_<项目主题>`

若日期尚未从 input 中抽取完成，则暂以占位符表示：

`<PERIOD>_<DOC_TYPE>_<CUSTOMER>_<PROJECT>`

### 3.5 检查规则

- 客户名称必须一致
- 项目主题必须一致
- 文档类型必须与输出类别匹配
- 文件名变量必须与正文关键字段一致

## 4. 当前可立即实施的规则

- 从输入文件名识别客户名称
- 从输入文件名识别项目主题
- 从输入文件名识别输入文档类型
- 从参考输出文件名提炼命名模式
- 将输出按 `文书型 / 表格型 / 检查型` 分层管理

## 5. 当前仍需补充的规则

- 从 PDF 正文提取日期区间
- 从 PDF 正文提取合同 / 订单关键条款
- 当前需求到底要输出哪些内部文档
- 不同内部文档类别的必填字段差异

## 6. 阶段结论

当前阶段已经可以开始：

- 模板拆解
- 变量建模
- 命名规则建模
- 检查规则建模

但还不适合直接宣布“自动转换规则已经定稿”。
