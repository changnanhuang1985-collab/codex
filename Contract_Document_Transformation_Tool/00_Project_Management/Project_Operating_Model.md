# 项目运营机制

## 1. 目的

本文档用于明确 `Contract Document Transformation Tool` 项目的运行方式、授权关系和汇报机制。

本项目从现在开始采用：

`CIO 治理放权 + 项目负责人日常推进`

的模式运行。

## 2. 角色分工

### Orion

`Orion` 作为 `changnan_codex` 级别的首席 CIO / 超级管理员，负责：

- 项目治理原则
- 工作方式校准
- 风险升级判断
- 阶段性 review
- 向用户输出阶段报告和最终报告

`Orion` 不负责本项目的日常逐项推进。

### 鞠

`鞠` 作为本项目负责人，负责：

- 日常推进
- 任务拆解
- 团队调度
- 产物落地
- 节奏管理
- 风险前移
- 向 `Orion` 汇报阶段结果

本项目后续默认由 `鞠` 出面推进。

## 3. 工作原则

- 用户不需要逐个管理本项目其他 AI 角色
- 用户主要向 `鞠` 下达项目执行任务
- `Orion` 只在治理、关键方向、节奏失衡或需要阶段报告时介入
- 高风险事项必须由 `鞠` 主动上报 `Orion`

## 4. 当前执行节奏

建议按以下节奏推进：

1. 样例治理
2. 模板拆解
3. 字段字典
4. 规则提炼
5. 检查与 review 设计
6. 第一轮工具实现验证
7. 阶段报告

## 5. 汇报机制

### 阶段报告

由 `Orion` 负责输出，内容包括：

- 当前阶段完成情况
- 已识别风险
- 下一阶段重点
- 需要用户确认事项

### 最终报告

由 `Orion` 汇总 `鞠` 的推进结果后输出，内容包括：

- 项目目标
- 实施路径
- 产物清单
- 验证结果
- 风险与限制
- 后续建议

## 6. 一句话说明

从现在开始，这个项目由 `鞠` 负责把事情做成，`Orion` 负责在关键节点给你交付清晰、可靠的报告。
